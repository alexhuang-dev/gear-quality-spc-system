# 面试讲解指南

这份文档是给你面试时直接使用的。你不需要把整个系统讲得特别学术，关键是让面试官迅速明白：你做的不是一个聊天机器人小玩具，而是一套有工程边界、有生产意识的质量分析系统。

## 30 秒版本

我做的是一个面向齿轮检测场景的质量分析系统。它把 CSV 检测数据转成可运行的 SPC 分析流程，用 Python 做确定性计算，用 LangGraph 做流程编排，用 FastAPI 提供接口，用 SQLite 保存历史批次，用 Harness 做结果一致性校验，还可以生成图表报告和告警载荷。Langflow 只是可视化展示入口，不是系统核心。

## 1 分钟版本

这个项目的核心思路是把“算数字”和“讲结果”分开。  
像均值、标准差、Cp、Cpk、Western Electric 判异、历史批次对比，这些都必须由确定性代码来做，不能交给大模型。  
所以我把核心能力放在 Python 后端，包括 SPC 计算、历史存储、图表生成、报告产出和 Harness 校验。  
在这个基础上，LangGraph 负责把计算、历史、告警、报告和校验串成可追踪流程，FastAPI 暴露成服务，Streamlit 提供看板，Langflow 作为演示入口去调用后端。  
这样做的好处是：系统既能展示，又不会让 LLM 改掉关键数值。

## 3 分钟版本

### 1. 问题背景

工业现场很多质量分析流程还是停留在人工看报表、人工做趋势判断，或者只是把 CSV 扔进 Excel。这样虽然能看结果，但不够自动化，也缺少历史追踪和稳定校验。

### 2. 我的架构判断

我把系统拆成两层：

- 确定性计算层：负责所有不能出错的部分
- 表达与展示层：负责解释、汇报、可视化和工作流演示

### 3. 技术实现

后端是纯 Python 生产链路：

- `FastAPI` 暴露分析服务
- `LangGraph` 做流程编排
- `SQLite` 保存历史批次
- `Streamlit` 做看板
- `pytest + golden cases` 做回归测试

核心能力包括：

- CSV 解析
- SPC 指标计算
- Western Electric 8 条规则
- 历史批次对比
- HTML / JSON / SVG 报告
- 可选 PDF
- Harness 一致性校验
- 告警 webhook 载荷生成

### 4. 为什么这不是玩具

因为它不是只会“生成一段分析话术”。

它具备真正工程化系统该有的东西：

- 确定性数值计算
- 历史可追溯
- 回归测试
- API 服务化
- 自动处理流程
- 图表和报告产物
- 告警接口
- 展示层与事实层分离

### 5. Langflow 在这里是什么角色

Langflow 是展示层，不是核心依赖。  
我保留它，是因为它非常适合演示流程和对外展示；但真正生产使用可以完全不依赖 Langflow，只用后端 Python 服务。

## 你可以主动强调的亮点

- 我没有把数值计算交给大模型，而是坚持用确定性代码做核心质量判断
- 我把历史对比、规则判异、报告生成、告警和校验做成了完整闭环
- 我在展示层保留了 Langflow，但把事实来源收敛到后端 API
- 我为系统补了 Harness 和 golden case 回归测试，不只是“跑通一次”
- 我把系统做成了可部署、可扩展、可迁移到真实工厂接口的结构

## 面试官可能会问的问题

### Q1：为什么要用 LangGraph？

因为这个项目已经不只是一个单函数脚本，而是一个包含计算、历史、报告、告警、校验的多阶段流程。  
LangGraph 更适合把这些节点编排成稳定的可追踪执行链，同时还能保留纯 Python 的确定性能力。

### Q2：为什么不用 LLM 直接分析全部流程？

因为质量分析里很多字段是不能幻觉的，比如 Cpk、控制限、规格限和历史差值。  
LLM 可以负责表达，但不能负责生成核心事实。

### Q3：为什么还保留 Langflow？

因为 Langflow 对演示、流程展示和业务沟通很友好。  
我把它定位成前台，把 Python API 定位成后台，这样两边优势都能保留。

### Q4：这套东西能直接进工厂吗？

软件架构上已经是生产导向的，但要真正落地到工厂，还需要三类外部条件：

- 真实规格限
- 真实数据采集接口
- 真实 MES / ERP / PLC 集成

## 最后一句你可以这么讲

这个项目最重要的不是用了多少框架，而是我明确区分了“什么必须由代码保证，什么可以交给模型表达”，所以它既能展示智能化，又不会失去工程可信度。
